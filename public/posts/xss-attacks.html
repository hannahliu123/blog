Watch Out: How Input Fields Caused My First Cross-Site Scripting (XSS) Attack

Hannah Liu

October 11, 2025

Recently, I experienced my first ever cybersecurity attack through the comments section of my blog. The attack involved a user typing code instead of plain text as input, which was then injected into my website’s HTML code. When the code ran, each of my blog posts were replaced by a blank page with the huge letters: "HACKED BY HATEYOU." At the time, I was terrified, but after looking into the issue, I realized that I had overlooked a very common security vulnerability: cross-site scripting.

Security was an area that I had never previously considered, as I was just starting to get into web development. But because of this cross-site scripting attack, I’ve learned to be more cautious of malicious users on the internet, and I hope this article can help you learn something new as well.

What is XSS?

Cross-site scripting, or XSS, is a web security issue that allows an attacker to insert malicious code into a website. XSS attacks typically involve manipulating some type of user input that isn’t properly processed or sanitized. Instead of typing normal text into an input field, attackers type code (usually Javascript) that is then embedded into the website and performs actions such as redirecting URLs, stealing user data, or replacing content on a webpage.

In the past, XSS has been an issue for companies like Google, eBay, Uber, WhatsApp, and more. In 2018, for example, an XSS attack on the British Airways was able to steal data from 380,000 booking transactions.

But in addition to those large-scale attacks, there is one type of a much simpler XSS attack that is often overlooked by beginners when creating any application that involves user inputs. This includes areas like comment sections, message boards, search bars, and setting up user profiles where a malicious user inputs code instead of plain text. Luckily, this problem can be easily resolved, as long as you are careful about how you implement and structure your code.

My First Experience

My first experience with an XSS attack was with my blog’s comment section. Before I fixed my code, I took submitted user input and directly embedded it into my comment section’s HTML code. Because of this, one user was able to exploit my mistake and add the following line of code that overrode my entire page with the words "HACKED BY HATEYOU".

<img src="x" onerror="document.body.innerHTML = '<h1>HACKED BY HATEYOU</h1>'">

At first, I was so scared that someone just deleted my website or stole some kind of information, but what actually happened was quite simple. Because I kept logs of each posted comment (I highly recommend you do that too), I was able to identify the comment that caused the error (which contained the line of code shown above). That line of code basically inserts an img tag with a faulty source and an onerror attribute that executes Javascript when an error (the faulty tag) occurs. The Javascript that executes ends up replacing my entire HTML document with the words "HACKED BY HATEYOU".

How to Prevent XSS

With simpler applications and input fields, the easiest way to prevent XSS attacks is to avoid using innerHTML with any areas involving user input. This is because when you insert text into your code using innerHTML, you are directly inserting the given input into the Document Object Model (DOM). Therefore, any code input will secretly run inside your website, likely causing malicious actions. Because of how common this security concern is, it is often recommended not to use innerHTML whenever dealing with inputs (as shown below).

One great replacement for innerHTML, which is what I personally used when I dealt with XSS attacks, is textContent. Instead of rendering HTML code, textContent will ignore all HTML tags and treat all strings as plain text (even if it has code embedded within it).

Another way to prevent XSS is to sanitize text input before it’s displayed on a screen by replacing special characters (< > & " ') with HTML character entities.

Another more tedious way to prevent XSS is to filter all input upon arrival. This is often a last-resort method when inputs get completely out of hand. It is more common for things like spam or inappropriate content, but for smaller sites, if you’re willing to go through all user input by hand, this could be a possible option. With spam being so common in comments sections recently, many blog owners have chosen to resort to this method in hopes of decluttering comment sections of unwanted content.

Lessons & Takeaways

This experience has been very eye-opening for me, as it was quite literally my first ever experience with a web security attack. I’m glad nothing serious happened, but thanks to this attack, I was able to realize how important security is on the Internet. My seemingly simple mistake has completely changed the way I view user input and security.

Most importantly, from this experience, I’ve learned that security isn’t just an issue for big companies. Attackers can target anyone building on the web, so it’s crucial to know how to prepare yourself for and prevent those attacks.
